# QY-EXEC 设计预算、长期演进约束与治理守则 (Architecture Budget & Long-term Governance)

本文件确立了 QY-EXEC 的系统上限、开发红线与治理原则，旨在防止系统在未来 5 到 10 年的持续迭代中发生“慢性功能膨胀”与“代码无序腐烂”。

---

## 1. 终极设计红线：不要让执行器学会“思考” (The Core Anti-Thinking Redline)

在 Agent 架构演进中，开发者最容易犯的错误是：**让执行层（Executor）变得越来越聪明，甚至承担部分的规划与决策。**
QY-EXEC 郑重确立此项**永久禁止红线 (Absolute Redline)**：

> **Executor 永远只负责执行，不负责目标推导，不负责任务分解，不负责长期记忆，不负责策略制定。**

* **红线细则**：
  * QY-EXEC 本身不参与任何 Prompt 提示词的动态切分与推导。它只是忠实地把底层物理工具的结果、状态和错误回传给上一层（Planner）。
  * 杜绝在执行端（Python）或协调端（Node.js）引入复杂的自主分支判断机制。
  * **系统稳定性的来源，不是每一层都很聪明，而是每一层都极致清晰地知道自己的边界，并极其确定、稳定地兑现职责。**

## 2. 设计预算限制 (Architecture Line-of-Code Budget)

为了防止系统在迭代中堆砌冗余辅助组件，导致单体代码体量过大产生认知负债与维护死区，我们设定以下 **代码量预算约束 (LOC Budget)**：

| 模块边界 | 最大 LOC 预算 | 超限纠偏策略 (Mitigation Action) |
| :--- | :--- | :--- |
| **QY-EXEC 核心执行器 (`executor.py`)** | `500 LOC` | 若超过此行数，必须剥离出专门的外部工具（如独立 script 文件）或安全核验类。核心轮询抢锁与调度逻辑必须极其轻量。 |
| **控制中心后端核心 (`server.ts`)** | `3000 LOC` | 控制面核心必须保持简练。若超过此行数，必须将业务路由（API Routes）拆解到 `src/routes/`，将核心后台 zombie 定时监测拆入独立 service 文件。 |
| **单一物理工具实现 (Tools)** | `300 LOC` | 物理工具（如 `patch` 引擎、`file_write`）必须保持单一职责（SRP）。如超过 300 行，必须拆分为核心库或底层 CLI 工具调用。 |

### 2.2 复杂度预算约束 (Complexity Budget)

除了行数限制外，任何新增或重构的业务逻辑均必须遵守以下 **复杂度红线 (Complexity Constraints)**：
* **最大代码块嵌套深度 (Max Nesting Depth <= 3)**：
  任何函数或控制分支（`if-else`、`for/while` 循环、`try-catch`）的嵌套深度绝对禁止超过 **3 层**。如果需要进行第 4 层嵌套处理，必须通过抽取辅助私有函数（Extract Method）强制打平，降低认知负荷。
* **最大函数行数 (Max Function Size <= 60 Lines)**：
  除了极少数涉及大字段映射的序列化函数外，任何单一函数的可执行代码行数（不计空行与注释）禁止超过 **60 行**。
* **禁止嵌套循环 (No Nested Loops)**：
  严禁编写“循环嵌套循环”（`for` 里面再套 `for`）的高阶时间复杂度算法。若有数据过滤与合并需求，必须在内存中使用 Map/Set 等哈希索引，在 $O(N)$ 空间与时间换取计算，保障执行效率。
* **圈复杂度上限 (Cyclomatic Complexity < 10)**：
  核心物理工具或状态轮询核心的分支因子数必须控制在 10 以内。复杂的多路分支必须退化为查表驱动（Table-driven design）或策略模式（Strategy pattern）。

## 3. 下一阶段关键治理机制规范 (Strategic Roadmap Guidelines)

为保障跨异构环境、长生命周期下的可靠性，QY-EXEC 后续架构在演进时必须遵循以下具体技术标准：

### 3.1 兼容性矩阵 (Compatibility Matrix Rule)
每一次控制面（Planner 适配端）或执行器进行重大升级，均必须在发布说明中主动声明或检测其 **Compatibility Matrix**：
* **对齐字段**：任务 JSON 的 `schemaVersion` 与执行端的 `executorVersion` 必须在启动心跳的第一毫秒进行对齐验证。
* **判定标准**：
  * 允许向下兼容（旧版 Executor 执行新版降级任务）。
  * 绝对禁止未知、未经测试的跨大版本混部运行。

### 3.2 链路追踪：全局关联 ID (Correlation ID Principle)
* **日志规范**：同一个物理任务的所有相关日志（包括：任务入队、任务抢占、执行心跳、模型调用、物理工具 stdout/stderr、僵尸回收、完成归档），必须在日志或 Event JSON 的头部统一携带唯一的 `correlation_id`（同 `task_id`）。
* **追溯价值**：严禁在未绑定关联 ID 的情况下输出分散的、不含上下文字段的调试文本。

### 3.3 状态反压与健康声明 (Backpressure & Health States)
* **状态定义**：心跳保活不等于系统 100% 处于可接收任务状态。
* **反压标准**：执行器在心跳包中必须携带明确的 `health` 状态：
  * `healthy`：健康且空闲，可接单。
  * `busy`：正在运行，队列已满，协调端自动执行 **反压（Backpressure）** 延迟分发。
  * `degraded`：处于内存或磁盘高水位预警，进入只读和自理维护。
  * `stopping`：收到优雅关闭通知，不再接受新任务抢占。

### 3.4 缺陷注入与混沌工程测试 (Failure Injection Rules)
系统的鲁棒性不来自于静态的“认为它安全”，而来自于**在残酷故障中反复熔断、重试、自救成功**：
* **混沌测试机制**：
  * 在自动化测试套件中建立混沌脚本（Chaos Injector），专门向正在运行的 QY-EXEC 主动、高频地注入各种极端故障：
    * 故意在任务执行时拔掉共享网盘网络。
    * 随机杀死物理执行器进程。
    * 在 SQLite 写入期间制造磁盘满额假象。
  * **通关标准**：在这种高强度的故障注入下，控制面 zombie 回收器必须能自动接管、计数自增、第 5 次完美熔断，不产生任何孤儿锁、僵尸运行文件、以及无法追溯的后台死锁进程。

## 4. 治理纪律：客观验证与严谨陈述

1. **去营销化、去玄学化描述**：所有系统架构、契约、SUCCESSOR 文档中，**严禁**使用夸大其词、自吹自擂的虚假广告或销售术语（如“绝对安全”、“完美无缺”、“军工级”）。
2. **用数字、代码与测试说话**：系统的核心防御能力（如路径白名单，防止穿越）必须有且仅能通过测试断言（Regression Tests）进行保证。凡是写在文档上的防护能力，必须在测试套件中存在针对性的、可自动跑通的攻击注入测试用例。
