# 启元理据之书 (The Book of Reasons)

> **历史序言 (Historical Intent)**
> 在漫长的软件演进中，代码是转瞬即逝的，甚至架构与协议也是可以重构的。
> 但一个文明之所以为文明，是因为其做决策背后的**根本理据（The Foundational Whys）**得以保留。
> 继承者在修改任何一行代码或重写任何一个模块前，必须翻开本理据之书，理解为什么当年的看护团队在各种充满诱惑的选择中，固执地选择了那条看似更笨、更重、但更确定的道路。
> **“不理解为什么这样设计，就不配拥有修改它的特权。”**

---

## 1. 为什么不用 Redis 或 RabbitMQ 作为主任务队列？
* **直觉诱惑**：使用 Redis/RabbitMQ 等专用消息队列可以提供微秒级的实时推送、天然的消息订阅机制、以及更低的网络开销，使系统看起来更像“企业级分布式微服务”。
* **文明裁决**：**拒绝。**
* **底层理据 (The Why)**：
  1. **零工具依赖的可观测性 (Zero-Tool Observability)**：任何运维人员无需配置、无需账号、无需运行命令行监控工具，仅凭最普通的文件管理器（Explorer/Finder）查看 `pending/`、`running/`、`completed/` 目录，就能在第 1 毫秒了解任务的积压、死锁、超时或异常状态。
  2. **物理级的安全隔离 (Air-gapped Physics)**：引入 TCP 消息队列会强制要求物理沙箱中的执行器保持对外网络端口的可达性。这会直接破坏“完全断网（Air-gapped）”的硬防护原则。通过局域网/网盘挂载盘轮询，执行层与调度层可以在网线物理拔除的情况下安全流转任务。
  3. **降低基础设施维护负债**：在低配工控机、单机边缘、或隔离的局域网环境里，部署并确保一个外部 Redis/RabbitMQ 进程 100% 活着，其运维开销和单点故障率远远高于最皮实的物理文件系统本身。

## 2. 为什么不用 SQLite 或其他关系型数据库存储状态？
* **直觉诱惑**：使用 SQLite 可以享受 SQL 语句查询的便利、完备的 ACID 事务特性，以及单文件易于搬迁的优势。
* **文明裁决**：**拒绝。**
* **底层理据 (The Why)**：
  1. **跨容器/跨网盘挂载锁兼容缺陷**：物理执行器在云端和边缘往往需要通过 NFS、Samba、CIFS 或 Rclone 挂载云盘。SQLite 底层严重的依赖 POSIX 级文件锁（POSIX Locking），但在网络挂载盘或虚拟文件系统中，这种锁极易发生一致性漂移或失效，导致数据库物理损坏（Database disk image is malformed）。
  2. **写冲突假死风险**：在多活调度端高频写同一数据库时，SQLite 的共享读写锁很容易触发 `database is locked` 报错，导致调度任务无征兆挂起。原子重命名（`os.rename`）在几乎所有异构操作系统与网络挂载协议中，都是被操作系统内核底层 100% 保证为物理原子（Atomic）操作的，具备更强的物理鲁棒性。

## 3. 为什么执行器（Executor - QY-EXEC）必须冷酷，绝对禁止学会规划？
* **直觉诱惑**：模型和工具发展越来越强，让执行端在面临“补丁失败、命令超时、目录不匹配”时，调用轻量级 LLM 进行自我修正、自我规划、甚至自我分解任务，不是能让执行端更聪明、成功率更高吗？
* **文明裁决**：**绝对禁止。**
* **底层理据 (The Why)**：
  1. **防止组织架构崩塌**：如果执行端学会了自我推导目标和分解任务，它就承担了上游 Planner 的职责。随着功能迭代，Executor 会逐渐膨胀为第二个 Planner，从而导致整个系统边界骨折，治理体系失效。
  2. **确定性优于智能性 (Determinism Over Intelligence)**：执行层是系统的“物理重力底座”。底座唯一的使命是提供 100% 可预测、可还原、无二次推理波动的物理反射。不聪明是执行器最大的美德。我们需要一个能够绝对确定执行指令的冷酷螺丝钉，而不是一个遇到问题会自我妥协和二次思考的规划者。

## 4. 为什么宁愿忍受轻微的轮询开销，也拒绝构建复杂的分布式锁？
* **直觉诱惑**：轮询物理文件系统每秒产生一次 I/O 损耗，在极端情况下可能发生几十毫秒的延迟。使用分布式协调中心（如 ZooKeeper、etcd、Consul）可以构建完美的分布式互斥锁。
* **文明裁决**：**拒绝。**
* **底层理据 (The Why)**：
  1. **简单优于复杂 (Simple > Complex)**：引入 etcd/Consul 会直接把系统的复杂度提高一个数量级，引入分布式脑裂（Split-Brain）、租约过期重写、时钟不同步等一整套极其难调试的复杂漏洞。
  2. **操作系统原生锁**：文件系统的重命名（`os.rename`）是底层的物理锁。当两个执行端同时试图把 `pending/task-1.json` 重命名到 `running/task-1.json` 时，只有一个能赢，另一个必败（抛出文件不存在或占用异常）。我们利用最朴素的操作系统特性，无情地平替了数万行分布式代码。

## 5. 为什么不提供跨任务的本地长期状态与缓存？
* **直觉诱惑**：如果执行端能缓存上一次任务编译产生的文件、或保留本地长期状态，下一次运行就能秒级加速，对用户体验不是更好吗？
* **文明裁决**：**拒绝。**
* **底层理据 (The Why)**：
  1. **隔离隐藏状态 (Zero Hidden States)**：长期本地缓存会在物理机上产生“隐藏状态”。时间一长，这个隐藏状态会因为磁盘满了、权限被篡改、文件损坏而发生系统性腐烂。
  2. **完美冷启动重放**：我们强制 Executor 每一次抢占任务都执行“冷启动”的彻底清理（无状态化）。这意味着任何一个任务被发往任何一台完全干净的新物理机，运行结果都能 100% 一致。我们宁可牺牲部分高频调用的速度，也要换取极端可靠的“无痕重放能力（Determinism in Playback）”。

---

## 6. 为什么安全规则（Policy）必须独立于具体工具代码热解耦运行？
* **直觉诱惑**：直接在 `bash` 工具的代码里面写正则拦截 `ping`/`nslookup`，或在 `read_file` 里面写 `if ".." in path`，实现起来快速且性能高。
* **文明裁决**：**拒绝。**
* **底层理据 (The Why)**：
  1. **防止策略泄漏与审计死角**：如果安全逻辑混在具体工具的业务代码中，每新增一个物理工具（如 Python 脚本、新 patch 模块），都必须由人工检查是否复制了安全规则。只要漏掉一个，系统防御立刻穿透。
  2. **集中审计与热升级机制**：将安全逻辑外置并集中在统一的前置过滤器（Policy Filter Chain）中，确保无论后续如何新增、重写或删除具体的物理工具，任何物理行为在发生前的第 1 毫秒都必须穿过此门槛，且策略的调整仅需修改配置文件，无需重新编译和发布物理代码，提供了真正的弹性防护。
