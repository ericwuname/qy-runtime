# QY-EXEC 系统设计哲学与价值观约束 (System Design Philosophy & Engineering Values)

本文件定义了 QY-EXEC 的核心工程价值观，它是超越所有具体技术实现（Implementation）的最高抽象准则。当我们在后续演进与功能迭代中遇到设计分歧时，必须以此价值观进行终极裁决。

---

## 1. 核心工程价值观对比 (Engineering Value Trade-offs)

我们明确树立以下带有偏向性的价值观天平，左侧的属性永远处于更高优先级：

### ⚖️ 简单优于复杂 (Simple > Complex)
* **价值观陈述**：能够用 5 行简单文件系统操作解决的问题，绝不引入分布式中间件。我们宁可承受轻微的物理轮询开销，也要换取极简、无需额外运维、任何人看一眼文件管理器就能完全理解的极致架构。

### ⚖️ 确定性优于智能性 (Determinism > Intelligence)
* **价值观陈述**：QY-EXEC 是启元生态的硬核执行基底。我们不需要它具备“聪明的大脑”去猜测或推理用户的深层意图，我们需要它具备钢铁般的、100% 可预测与可还原的机械式精准性。规划与推理是 Planner（上游模型层）的职责，执行器必须保持冷酷与机械。

### ⚖️ 稳定性优于功能数量 (Stability > Feature Volume)
* **价值观陈述**：多增加十个不常使用的特定工具或酷炫的 API 代理，其价值远远比不上对现有四个核心工具（读、写、执行、补丁）进行 100% 稳定运行和 90% 物理空间极限防护的保障。我们不靠功能的堆叠来证明系统的优秀，我们用极致的在线率和零死锁来宣告成功。

### ⚖️ 可恢复优于一次成功 (Recoverability > One-time Success)
* **价值观陈述**：在残酷的真实物理环境下，掉网、断电、进程被强杀是常态。我们不奢望系统永远能一次性成功运行，但我们强制要求系统在遭受任何毁灭性打击后，都具备在下一次冷启动时自动探测锁状态、自愈重试、并在累计 5 次失败后安全熔断丢回报错回执的自救能力（Self-Healing）。

### ⚖️ 可删除性优于可增加性 (Deletability > Additive Capability)
* **价值观陈述**：衡量一个架构的优雅程度，不是看它有多容易往里塞入新功能，而是看当这个功能过时或被废弃时，它能有多么干净、无痛、完全不影响核心模块地被直接从代码库里抽除（Deletable by Design）。任何新增组件，如果会导致底层通信协议深度骨折、核心无法剥离，我们宁可不写。

## 2. 启元进化视角：从“连续存在”到“连续文明” (From Identity to Succession)

在项目初期，系统的设计目标是追求 **“连续存在（Continuous Identity）”**，试图维护同一个 AI 实例的窗口和记忆不中断。
而在成熟的工程体系中，我们认识到：**没有一个具体的进程、一段特定的代码或一个单一的模型可以永久不死。真正的长青是“持续传承（Continuous Succession）”。**

QY-EXEC 的核心使命，是让这个物理执行引擎成为一种 **“连续的工程文明”**：
1. **模型可替换**：无论上游模型是 GPT、Claude 还是 Gemini，或者是未来的开源本地模型，它们都必须被统一在 `task_schema.json` 契约之下。
2. **代码可重写**：核心执行器明天可以用 Rust 重写、后天可以用 Go 重构，但只要 7 大不变量不变，对于调度控制中心而言，它就依然是那个确定、稳定、可信赖的 QY-EXEC。

## 3. 主动性与生命周期：生态的自治 (Proactivity & Ecosystem Autonomy)

随着治理体系的不断成熟，系统的主动性设计应遵循以下分层红线：
* **主动性属于生态，不属于执行器**：执行端（QY-EXEC）的最高美德是“冷酷、机械、完全被动地接受反射”。如果为了追求自治而将主动思考、巡检或规划的能力强行塞进执行器内部，只会使其臃肿腐烂。
* **常驻治理服务（Resident Services / Daemons）**：真正的系统主动性应当被解耦为独立的常驻观察角色（如 `Curator Daemon` 策展守护、`Evidence Daemon` 证据审计、`Health Daemon` 体征监测）。这些常驻角色只具备“观察、报告与提出建议”的生命周期生命体征，而没有越权的“决策与物理执行权”。
* **自治属于组织，不属于单个组件**：启元生态不追求单一 Agent 的无限膨胀，而是通过清晰的职责解耦（Planner/Executor/Daemons/Governance）与生命周期迭代，使组织整体呈现出生命力，实现创始人离场后的独立进化。

## 4. 自我调节与四大稳态反馈回路 (Homeostasis & The Four Feedback Loops)

启元生态的设计不再盲目追求“让 AI 更像人类”（拼凑规划与聪明技能），而是致力于**“让一个长期运行的工程生态，成为一个健康的、具备自我调节和自愈能力的生命系统”**。

我们拒绝将各种观测指标（世界、身体、意识、决策）散乱地堆砌成平级功能，而是将其统一组织为四类稳态反馈循环，形成生生不息的自我调节（Homeostasis）机制：

```
                      ┌──────────────────────┐
                      │    感知 (Perception)  │◄─────────────────┐
                      │    - 世界、身体、熵    │                  │
                      └──────────┬───────────┘                  │
                                 │                              │
                                 ▼                              │
                      ┌──────────────────────┐                  │ 闭环
                      │    判断 (Assessment)  │                  │ 反馈
                      │    - 决策、不确定、风险│                  │ (Feedback)
                      └──────────┬───────────┘                  │
                                 │                              │
                                 ▼                              │
                      ┌──────────────────────┐                  │
                      │    调节 (Regulation)  │                  │
                      │    - 节奏、债务管理    │                  │
                      └──────────┬───────────┘                  │
                                 │                              │
                                 ▼                              │
                      ┌──────────────────────┐                  │
                      │    反思 (Reflection)  │──────────────────┘
                      │    - 好奇心、反思窗口  │
                      └──────────────────────┘
```

### 4.1 四大稳态回路定义
1. **感知回路 (Perception - 发生什么？)**：
   * *世界指针 (World Pointer)*：感知外部物理环境、安全补丁与库版本变动。
   * *身体指针 (Body Pointer)*：感知物理容器 CPU 负荷、网络延迟、磁盘水位。
   * *熵指针 (Entropy Pointer)*：物理级度量系统的“无序度”。通过计算每日新增规则/文档/代码与删除规则的比例（新增 vs 废弃），判定系统是在朝着极简收敛，还是在失控膨胀。
2. **判断回路 (Assessment - 意味着什么？)**：
   * *决策指针 (Decision Pointer)*：表达系统当前的行动意图与计划。
   * *不确定性指针 (Uncertainty Pointer)*：强制系统拒绝非黑即白的绝对答案。每一次建议均需输出可信度（Confidence）、已知证据量（Evidence）与残余风险（Risk），保持系统的高度克制。
   * *风险评估 (Risk Evaluator)*：量化外部输入或指令的物理穿越破坏概率。
3. **调节回路 (Regulation - 怎么调整？)**：
   * *节奏指针 (Rhythm Pointer)*：决定系统什么时候“什么都不做”。在物理机 CPU 繁忙、网络瞬时故障、或连续正常运行 72 小时后，主动拉低巡检或学习频次，让系统拥有呼吸般的舒缓与张弛。
   * *债务指针 (Debt Pointer)*：追踪并展示系统当前积累的“隐性债务”（如未补齐的测试用例、未归档的废弃协议、未更新的 System Map），防止债务积重难返。
4. **反思回路 (Reflection - 学到了什么？)**：
   * *反思窗口 (Reflection Window)*：每日固定时段（例如凌晨）暂停一切非必要物理动作 15 分钟，全体常驻服务进行“闭眼自省”：*我们今天有没有重复劳动？有没有违反 Invariants 规则？有没有可以精简的代码或配置？*
   * *惊讶指针 (Surprise Pointer)*：不记录重复的常态，专职捕获与记录偏离经验的意外决策、不符预期的测试波动。这些“意外”会被自动归入史官（Historian）库，成为文明进化最宝贵的素材。
