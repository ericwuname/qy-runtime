## 0. 第一性原理与定位价值 (First Principle & Vision)

QY-EXEC（QiYuan Executor）的愿景定位为 **“安全可靠的物理级离线 Agent 执行容器与双向数据网桥”**。
若未来的演进路线由于资源约束必须做出取舍，我们的核心优先级排序（North Star Priority）如下：
1. **安全沙箱隔离性 (Security & Isolation)**：极尽一切手段，在最底层的物理/虚拟执行端，对 Agent 指令、网络出站、旁路侧信道建立完全熔断防护。
2. **高可靠容灾与确定性 (Reliability & Fault-Tolerance)**：保障在不可靠局域网/网盘、极端进程死锁环境下，系统状态永远可审计、可恢复、不丢失。
3. **生态兼容与多模型协同 (Compatibility & Flexibility)**：保持对标准协议的无缝支持。

## 1. 架构层：双向桥接的“最终一致性”设计

### 1.1 文件系统同步的延迟窗口量化 (Sync Latency Constraints)
在当前的 MVP (v1.0) 阶段，QY-EXEC 的核心桥接逻辑建立在以下物理与拓扑假设之上：
* **局域部署假设**：协调器（Node.js Control Plane）与物理执行器（Python Executor）部署在 **同一台物理/虚拟机器** 或 **超低延迟的局域网络 (LAN)** 内。
* **延迟窗口阈值**：共享文件系统同步延迟默认限制在 **< 100ms**。在此窗口内，文件锁抢占与状态轮询基本属于实时同步级别。
* **跨地域高可用演进 (v2.0)**：一旦未来升级至跨地域或多节点云部署（如上海协调器对北京执行器），文件系统同步可能退化至秒级甚至分钟级延迟，产生不可预测的“最终一致性”裂缝。届时，系统将：
  1. 引入 **Redis Pub/Sub** 或 **RabbitMQ** 等高性能消息队列作为核心的实时信号调度控制面通道。
  2. 将原有的物理文件系统目录（`pending/`, `running/`, `completed/`）降级为 **非实时持久化存储与审计日志层**，通过信号流驱动任务抢占，消除一致性断层。

#### 💡 v2.0 消息队列的“最终一致性与语义鸿沟”应对：
消息队列通常仅支持 **“至少一次 (At Least Once)”** (RabbitMQ) 或 **“至多一次 (At Most Once)”** (Redis Pub/Sub) 交付语义，而分布式文件系统桥接通常需要 **“恰好一次 (Exactly Once)”** 语义。
为了弥合这一语义鸿沟，QY-EXEC 设计如下：**消息队列仅充当“唤醒信号 (Wake-up Alert)”，不承载具体任务报文与状态。**
执行器收到唤醒通知后，必须依然通过文件系统的原子锁重命名（`os.rename` 抢占）完成任务分配。这种“信号+锁拉取”的混合同步机制确保了消息丢包或重复通知仅会影响实时响应时间，绝不会引发状态错乱。

### 1.2 "共享存储" 单点故障容灾策略 (SPOF Mitigation)
为了避免共享存储（网盘、Rclone 挂载、Syncthing 同步目录）由于挂载丢失或网络中断导致系统雪崩（如无法读取、心跳假死导致重复回滚），QY-EXEC v1.1 采用了 **“本地多级缓存 + 异步合并”** 的容灾机制：
1. **多级持久化缓存**：协调器在本地磁盘维护完整的内存副本 `activeTasks` 与标准事务数据库 `tasks.json`。
2. **熔断与优雅降级**：
   * 当检测到底层共享文件路径不可写、物理挂载丢失或同步网络发生断开时，协调器自动熔断与同步目录的物理写入动作，将操作退化至本地 `tasks.json`。
   * 期间新创建的任务在本地标记为挂起状态。
3. **最终合并对齐**：当共享挂载点网络恢复并重新上线后，同步轮询线程会执行单向幂等重建，恢复状态。

#### ⚠️ 消息队列组件的非核心降级设计 (Optionality Guarantee)
在 v2.0 时代，消息队列被定义为 **非核心依赖的可选优化组件**。系统保留完整的底层文件目录轮询扫描作为兜底。若消息中间件发生瘫痪（如 Redis/RabbitMQ 宕机或网络中断），控制面与执行端能自动、无感地退化至高频文件轮询模式，从而保障系统在运维断档期依旧具备极高鲁棒性。

## 2. 安全层：侧信道攻击防御体系 (Side-Channel Hardening)

除了 CPU/内存上限等常规 Job Object 硬件限制外，QY-EXEC 还对以下隐蔽的侧信道出站及破坏路径实施了硬性熔断：

### 2.1 DNS 隧道数据外泄拦截 (DNS Tunneling Prevention)
* **攻击场景**：Agent 绕过应用层防火墙，通过执行 `nslookup`、`ping` 等工具将敏感数据（如 API Key 碎片、私钥）进行 Base64 编码，作为子域名查询发送至攻击者控制的恶意外部 DNS 服务器（例如 `base64-encoded-secret.attacker.com`），从而实现数据出站泄露。
* **防御手段**：
  * **硬性拦截规则**：将 `nslookup`, `ping`, `tracert`, `nbtstat`, `dig`, `host` 等经典域名查询与网络排障指令加入 **系统硬性黑名单**（已由 `DANGEROUS_PATTERNS` 代码级硬编码管控），默认拦截。
  * **显式审计授权**：在非必要调试场景下，此类命令一旦被触发，即视为高危安全事件并上报安全审计日志（`security.log`），阻止任务继续流转。

### 2.2 电源管理与系统级破坏防护 (Power Management Hardening)
* **攻击场景**：Agent 试图执行系统电源控制、休眠、停机或改变底层核心服务的运行级别，使宿主机物理关机，导致全系统僵死。
* **防御手段**：
  * **指令黑名单拦截**：在 `DANGEROUS_PATTERNS` 黑名单中加入了针对 `powercfg`, `shutdown`, `reboot`, `stop-computer`, `restart-computer` 以及调用电源管理动态链接库 `rundll32.exe powrprof.dll` 的全词精准正则拦截。
  * **零特权运行原则**：严格禁止以 System 或 root 权限启动 Python 执行器，防止 Agent 突破黑名单通过特权直接下达电源中断指令。

### 2.3 物理侧信道与时序侧信道的边界界定 (Timing & Physical Side-Channel Limits)
* **2.3.1 时序侧信道 (Timing Side-Channels)**：
  * **场景**：Agent 通过反复测量常规安全指令（如 `dir` 敏感目录或 `type` 文件）在耗时上的微小差异，推断出文件的真实大小、内网网关拓扑结构或外部服务的可达状态。
  * **已知局限**：本系统在 MVP (v1.0) 阶段默认不防御应用层时序旁路。如部署环境极其严苛且涉及保密性要求极高，应当将执行器独立配置于专有硬件沙箱中，或对基础 I/O 操作时延进行均匀化加噪补零。
* **2.3.2 电磁/声学与功耗侧信道 (Acoustic, Electromagnetic & Power Side-Channels)**：
  * **场景**：外部敌对实体通过监控计算设备在加解密操作或键盘输入时的电磁波辐射、功耗波动波形、甚至是散热风扇与硬盘声学震动，还原核心机密。
  * **已知局限**：QY-EXEC 作为软件层网桥架构，不对物理/硬件级电磁旁路信道负责。若物理环境面临此威胁等级（如军工、绝密物理战机），建议将运行载体强制置于 **法拉第笼（Faraday Cage）** 或通过 **TEMPEST 级别认证** 的高安硬件套件中运行。

## 3. 运维与生命周期管理 (Operations & Lifecycle)

### 3.1 任务 ID 冲突规避设计 (Multi-Instance Collision Avoidance)
为保证在多实例、多活协调器部署场景下的唯一性，避免因时钟回拨或高并发导致的任务 ID 冲突，QY-EXEC 设计了包含“运行上下文特征”的 ID 编码规范：
* **编码公式**：`task-${Timestamp}-${Hostname}-${ProcessID}`
* **实例保证**：将时钟毫秒数、宿主机 Hostname 的字母数字指纹以及协调器的进程 PID 进行多重拼接。这样即便在多个分布式协调器容器并发抢占时，依然能保证 ID 的全局唯一性。

### 3.2 日志冷存储与留存归档策略 (Log Archiving)
为防止长期高频任务执行产生巨量 JSON Lines 日志撑爆系统磁盘，或因简单日志文件轮转（Log Rotation）被无情抹除导致长期安全审计证据链丢失：
* **归档机制**：
  * **温数据轮转**：当前日志达到 10MB 阈值时触发自动更名轮转，本地保留最多 5 个近期热温日志文件。
  * **冷存储策略 (v1.1)**：轮转淘汰的日志在移出热区时自动压缩为 `.log.gz`，存入特定的 `logs/archive/` 目录，维持 90 天的最短保留周期。
  * **远程备份**：通过外部同步任务（如 Rclone Cron 任务）每 24 小时增量同步 `logs/archive/` 目录至低成本云存储中。

### 3.3 世代循环熔断机制 (Generation Breaker Limit)
* **问题描述**：为应对僵尸任务，系统会周期性自动回收心跳过期的物理任务，并使该任务的 `generation` 与 `retryCount` 递增、将其掷回待挂起状态。若由于物理执行器底层逻辑死锁或环境网络不稳定导致任务持续挂起、挂起后又反复死锁，会引发 `generation` 无限制无界增长，产出冗余文件碎片并加剧系统空转消耗。
* **熔断方案 (v1.0.1 - 已代码化实现)**：
  * **重试限额**：系统硬性规定最大僵尸回收重试上限为 **5 次** (`retryCount >= 5`)。
  * **熔断控制**：当任务第 5 次心跳超时且无法恢复时，自动回收监测器拒绝将任务移回 `pending/`，直接激活安全熔断开关：
    1. 将内存中对应任务状态标记为 `failed`。
    2. 生成包含故障追溯原因的 `_result.json`，投递至 `completed/` 以便调用端及时报错（绝不无限等待）。
    3. 清理物理残留的 `running/` 文件以及 `.heartbeat` 锁，发布 CRITICAL 系统日志，进入待审计流程，等待人工排障介入。

## 4. 演进路线图：协议解耦 (Roadmap: Vendor-Neutral Protocol)

QY-EXEC v2.0 将重点研究 **与模型服务商特定协议解耦** 的中间架构层，以应对未来 OpenAI Function Calling 等标准接口协议的大幅改动风险：

* **核心目标**：实现一套不依赖任何特定模型厂商（OpenAI, Google Gemini, Anthropic Claude）原生工具调用格式的 **中间通用动作描述协议 (Unified Action Representation, UAR)**。
* **架构设计**：
  1. **适配层 (Model Adapters)**：各类云端模型只负责将输入自然语言翻译成标准的 UAR JSON（定义如：`{ "action": "file_write", "params": {...} }`）。
  2. **执行引擎 (Neutral Engine)**：物理执行器只监听、验证并执行 UAR 标准指令，屏蔽底层所有厂商协议的细节变化，彻底规避 Vendor Lock-in 与长期认知负债。
