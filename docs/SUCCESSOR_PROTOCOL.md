# 启元执行器继承者移交协议 (QY-EXEC Successor Protocol)

> **前言 (Onboarding Preface)**
> 无论你是由于主程轮换、项目交接，还是在紧急情况下接手本系统，请记住：**QY-EXEC 是物理级别的指令执行通道。**
> 在对本系统进行任何修改或生产环境启动前，必须完全理解本协议载明的手续、架构权衡以及已知极限。安全与高可靠性是系统的终极准则。

---

## 1. 快速上手：系统启停标准步骤 (Start & Stop Operations)

QY-EXEC 采用 Node.js (Vite + Express) 作为控制中心，物理端采用 Python (Job Object) 守护执行器。

### 1.1 系统启动步骤
1. **检查依赖环境**：
   * 确保本地安装有 Node.js v18+ 与 Python 3.10+。
   * 物理执行器运行平台建议为 Windows Server / Windows 10（以便完全调用 Windows Job Object），或在带有 seccomp 限制的 Linux 容器内。
2. **初始化配置文件**：
   * 确认根目录下存在合规的 `ai_config.json` 与 `.env` 文件。
3. **拉起服务进程**：
   * 开发模式：执行 `npm run dev` 启动 Express 控制面服务。
   * 物理执行端：启动 `bridges/qi_yuan_executor/` 下的 Python 守护进程轮询。
4. **验证健康状态**：
   * 请求 `GET http://localhost:3000/api/health` 确认控制面正常返回。

### 1.2 系统停止与熔断步骤
1. **优雅下线控制端**：
   * 向 Node.js 进程发送 `SIGTERM` 信号。系统会自动处理挂起任务，拒绝新入队任务。
2. **物理回收执行端**：
   * 物理执行器线程接收优雅关闭通知，执行完当前正在处理的原子指令后退出，同时移除 `running/` 中的物理文件与 `.heartbeat` 标记。
3. **极端状态强行熔断**：
   * 若执行端发生硬件卡死，请先强制杀掉 Python 子进程。
   * 控制端背景 zombie 监视器在 60s 内未收到心跳将自动执行回收，若重试 5 次失败，则启动系统熔断器强行下线该任务。

## 2. 认知指南：关键配置参数的深度权衡 (Configuration Trade-offs)

各关键参数经过精细化生产级博弈微调，非极特殊排障场景**切勿随意更改**：

| 配置参数 | 默认值 | 合理范围 | 权衡决策与调优方向 (Trade-offs) |
| :--- | :--- | :--- | :--- |
| **心跳超时判定间隔** | `60s` | `15s - 180s` | **缩短**能更灵敏地感知执行端假死并快速回收重试，但若物理机出现 I/O 抖动或网络瞬变，极易导致高频**误判**并产生任务重试雪崩；**延长**可降低磁盘 I/O 压力，但会让系统卡死感知钝化。 |
| **僵尸重试上限** | `5` | `3 - 10` | **限制 generation 无界增长的核心卡点**。设置为 `5` 次能留有足够的临时网络波动弹性；若超过 `5` 次持续假死，系统将判定执行器存在无法自我恢复的死锁/崩溃，必须强制熔断报错，等待人工介入。 |
| **日志自动轮转阈值** | `10MB` | `2MB - 50MB` | 控制温数据日志文件的单体占用。设置为 `10MB` 能够避免高频写盘引起的性能损耗，同时确保单文件在文本编辑中可秒级开启；搭配最多 `5` 个保留历史，防磁盘撑爆。 |
| **低磁盘空间熔断水位线** | `90%` | `85% - 95%` | **P1 自救保护罩**。由于执行器在写文件、执行模型任务时会生成大量临时字节，一旦可用空间低于 10%，将极易导致底层文件锁崩溃、SQLite写失败等致命灾难。故在 90% 时强制硬熔断，安全降级。 |

## 3. 继承者避坑：已知系统局限性清单 (Known Constraints)

作为第二任看护者，不要试图在没有深刻理论准备的情况下通过修改代码来“解决”以下问题，它们属于系统的**已知设计局限性 (Known Constraints)**：

1. **时序侧信道不防御**：
   系统不防范通过测量指令耗时微小偏差（如 `dir` 的响应纳秒差）推算文件存在性或网络拓扑的攻击。如部署于绝密环境，请在专用隔离物理沙箱内运行。
2. **物理级（电磁、声学、功耗）侧信道不防御**：
   软件层网桥无法在硅片与波形层面屏蔽电磁波辐射。如面临此威胁，请将硬件移入法拉第笼（Faraday Cage）。
3. **消息队列最终一致性延迟**：
   高并发下，由于网络拓扑的不同，桥接网盘在跨节点同步中可能会出现秒级延迟。系统已采用“信号唤醒+文件原子重命名抢占”来对冲该延迟。任何对锁文件的优化都不可破坏 `os.rename` 的原子性保证。

## 4. 第一接手者应做的三件事 (The First Three Tasks)

若你刚刚拿到此系统的控制权，请立即按顺序执行以下三件事，以验证你是否完全掌控了系统：

1. **执行一次人工僵尸任务回收审计**：
   故意制造一次执行端进程挂起 65 秒以上的事件，登录控制台观察日志是否正确输出 `[ZOMBIE_RECOVERY]` 警告，并确认任务的 `generation` 与 `retryCount` 正常自增 1，且被重新丢回 `pending` 状态。
2. **触发一次极限重试，验证 Generation Breaker 的威力**：
   让物理端持续假死，使得重试计数累积。在第 5 次超时后，确认系统是否自动生成了以 `_result.json` 结尾的失败报告，且成功自动解除了 `running/` 中的锁。
3. **阅读 `/docs/contracts.md` 契约文档**：
   理清与 `AGENTS.md` 及控制面板的数据流动关系，避免因修改对外接口字段导致前端组件解析挂起。
