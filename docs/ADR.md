# QY-EXEC 架构决策记录 (Architecture Decision Records - ADR)

本文件详细记录了 QY-EXEC 在演进历程中的重大技术架构决策（Why & Why Not），作为继承者与看护团队理解系统设计权衡的唯一官方历史底座。

---

## ADR-001: 采用基于文件系统 (File-Queue) 的调度机制

* **状态**：已接受 (Accepted)
* **创建时间**：2026-07-13
* **决策上下文**：
  在系统立项之初，需要确定协调器（Control Plane）与物理执行器（Executor）之间的任务流转协议。备选方案包括：
  1. 采用 **Redis / RabbitMQ** 作为实时消息队列与状态同步层。
  2. 采用 **SQLite / SQLite WAL** 等关系型嵌入数据库作为共享持久化媒介。
  3. 采用 **纯文件目录系统（Shared File System Queue）** 通过原子重命名（os.rename）进行排队抢锁。
* **决策理由 (Why File-Queue?)**：
  * **极限可观测性 (Zero-Tool Observability)**：任何运维人员或开发者无需安装特定数据库查看工具或命令行队列监视器，只需打开本地文件管理器即可通过 `pending/`、`running/`、`completed/` 目录一眼看清当前任务的所有积压、死锁、超时与运行状态。
  * **无外部依赖的极致解耦**：物理执行器只需监听共享网盘/挂载网盘的本地目录，即可实现跨越容器沙箱、异构主机的任务拉取。无网络协议栈依赖，极大降低了由于网络端口、防火墙配置引起的不可达风险。
  * **原子重命名天然锁**：Unix/Windows 文件系统支持原子级重命名（`os.rename`）。两个执行器进程同时试图移动同一个文件时，底层操作系统能天然提供强一致性的排他性保证，防止任务被重复调度，免去了编写复杂的分布式互斥锁（Distributed Mutex）的开销。
* **为什么不采用其他方案 (Why NOT Alternatives?)**：
  * **为什么不用 Redis / RabbitMQ 作为主调度队列？**
    * *单点硬件依赖与部署开销*：引入 Redis/RabbitMQ 会强制要求宿主机或容器集群常驻该中间件服务。在低配单机、局域网单机或边缘物理部署（如离线 Windows 终端）时，维护外部服务的运行状态会显著增加运维开销与崩溃概率。
    * *网络端口暴露与隔离失效*：使用外部网络队列会强制要求物理执行沙箱开启对外网络出站端口来访问队列实例，直接打破了“网络全断开（Air-gapped）”的安全沙箱隔离原则。
  * **为什么不用 SQLite / SQLite WAL 作为事务介质？**
    * *锁竞争死锁风险*：在多物理执行进程/多活调度控制端同时写同一个 `.db` 文件时，SQLite 的共享文件锁易引发 `database is locked` 异常，进而造成多进程执行冲突假死。
    * *跨容器挂载兼容性缺陷*：在一些分布式挂载文件系统（如 NFS、CIFS、Rclone 挂载网盘）中，SQLite 底层所依赖的 POSIX 锁无法提供跨网络虚拟挂载的原子一致性保证，极易导致数据库索引物理损坏（Corrupted Database）。
  * **为什么不用 Windows Message Queue (MSMQ) / DBus？**
    * *平台强绑定锁死*：MSMQ 在现代 Windows 系统中已默认废弃并降级，且在 Linux/Unix 容器集群下完全不可用。为了保留后续跨平台运行的兼容性，绝对禁止引入专属于单一操作系统的进程间通信（IPC）机制。
* **妥协与风险 (Consequences & Trade-offs)**：
  * *高频磁盘 I/O 损耗*：轮询物理文件系统会产生持续的读写损耗。系统已在配置中引入 10 秒心跳保活和 1 秒任务扫描间隔来平抑 I/O 尖峰。
  * *跨地域网盘延迟*：若挂载盘在跨国或跨地域同步中产生大于 200ms 的延迟，可能发生短暂锁冲突。我们已规定在 2.0 时代将引入消息队列仅充当“唤醒信号”，底层仍保留原子重命名抢锁。

## ADR-002: 物理隔离黑名单（Deny-list）与白名单路径的静态本地控制

* **状态**：已接受 (Accepted)
* **创建时间**：2026-07-13
* **决策上下文**：
  针对 Agent 执行任意系统命令带来的安全隐患（如 DNS 隧道泄密、执行系统电源破坏、越权读写系统敏感文件），需要确立一套绝对、高可操作性的安全沙箱。
* **决策理由**：
  * 拒绝外部规则推导，坚持 **“本地硬核静态防护机制”**。
  * 结合黑名单（精确正则拦截 `nslookup` 等 DNS 查询与电源控制命令）与绝对路径解析白名单（拦截所有 `..` 相对穿越逃逸），从底层系统 API 调用端和文件读写入口链直接拦截。
  * 这确保了即使上游 Planner（LLM）由于遭受注入攻击被诱导产生恶意行为，QY-EXEC 的执行关卡也能提供不可妥协的物理熔断（Fail-Safe）。
* **妥协与风险**：
  * 开发者在执行合法系统运维任务（如合法内网 `ping` 测试）时可能触发黑名单，需在特定配置文件中通过白名单身份（submitter）静态绕过。

## ADR-003: 提交者身份声明白名单机制 (Submitter Whitelist)

* **状态**：已接受 (Accepted)
* **创建时间**：2026-07-13
* **决策上下文**：
  随着多实例和多级 Agent 生态的建立，任务提交源不再局限于人类 CEO。恶意 Agent 如果绕过鉴权直接投递非法任务，将严重危害物理主机的安全。
* **决策理由**：
  * 为 API `/api/tasks` 接口设立 `allowedSubmitters` 静态白名单策略。
  * **身份隔离**：禁止未知实体将任务塞入 pending 队列，在数据流入协调器的第一时间进行 403 阻断并记录审计日志。
* **妥协与风险**：
  * 当前阶段处于身份声明（Identity Claim）鉴权。若未来升级到跨网络多实例，该声明存在被篡改、伪造的可能。届时，此 ADR 将升级为“基于 JWT / 非对称密钥签名的身份验证协议 (Cryptographic Signature ADR)”。
