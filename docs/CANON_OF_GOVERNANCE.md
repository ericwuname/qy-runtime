# 启元治理典章与终局遗产协议 (The Canon of Governance & Legacy Protocol)

> **法典序言 (The Canonical Intent)**
> 任何进入繁荣期的工程文明，其终极挑战不再是“如何设计新规”，而是**当规则、决策和历史发生冲突时，如何进行终极裁决；以及当文明走向终局时，如何为后世留下干净的遗产。**
> 本法典定义了启元体系的终极解释权级、例外豁免程序、失败博物馆规范、监督者轮值与退役制度，以及系统停止演变时的《遗产冻结协议》，保障启元自治架构在长周期运行中的绝对逻辑收敛。

---

## 1. 解释裁决权优先序 (The Interpretation Rule)

在数年乃至数十年的物理演进中，当哲理、基因不变量、自动化证据、历史决策与现实物理受限发生不可调和的冲突时，系统禁止进行无休止的人工争论。我们确立以下 **“神圣解释优先序”** 进行终极裁决（越往上优先级越高）：

```
                  ┌────────────────────────┐
                  │ 1. Reality (物理现实)  │ ─── 终极重力 (物理不可违背，如磁盘满了)
                  └──────────┬─────────────┘
                             │
                             ▼
                  ┌────────────────────────┐
                  │ 2. Invariants (基因)   │ ─── 永恒底线 (如 Executor 永不规划)
                  └──────────┬─────────────┘
                             │
                             ▼
                  ┌────────────────────────┐
                  │ 3. Evidence (证据)     │ ─── 自动化测试证据证明高于一切纸面承诺
                  └──────────┬─────────────┘
                             │
                             ▼
                  ┌────────────────────────┐
                  │ 4. Constitution (宪法) │ ─── 演化、极简、与治理典章
                  └──────────┬─────────────┘
                             │
                             ▼
                  ┌────────────────────────┐
                  │ 5. ADR (历史选择)      │ ─── 历史上的架构决策理由
                  └──────────┬─────────────┘
                             │
                             ▼
                  ┌────────────────────────┐
                  │ 6. Philosophy (价值观) │ ─── 设计倾向与叙事文化
                  └────────────────────────┘
```

* **裁决释义**：
  * **物理现实高于一切**：如果物理机器由于磁盘溢出、硬件断电导致系统停机，任何“不变量”都必须让位于《移交协议（SUCCESSOR_PROTOCOL.md）》中的紧急熔断自救（Zombie Reclaim）机制。
  * **测试证据高于制度规范**：如果某项不变量（如 Executor 彻底无状态化）在纸面上写得很完美，但自动化混沌测试（Evidence）检测出了 Yellow 警报，系统判定该不变量目前处于“未落地/失效”状态，必须立刻阻断 PR 合流，拒绝空口承诺。
  * **历史服从当下宪法**：当两年前的 ADR 选型依据与当下的《演化宪法（EVOLUTION_CONSTITUTION.md）》发生冲突时，以当前宪法为最高判准，并通过撰写新的 ADR 宣告旧 ADR 废止（Superseded）。

## 2. 例外豁免程序 (The Exception Protocol)

我们不追求死板的教条主义。当物理环境发生革命性剧变（例如未来网络传输速度或存储介质发生维度级突破，导致 File Queue 确实成为系统唯一的物理瓶颈时），系统允许违反或修改不变量，但必须履行 **“摩擦性例外程序”**：

1. **例外申报 (Exception Proposal)**：发起者必须书写专门的 `Type D` 提案，详细陈述物理边界的变化。
2. **阻力听证 (Friction Review)**：治理层在多活沙箱中执行不低于 100 小时的灰度对比验证，客观度量“违反该原则换取的性能，与可能产生的治理债之间的比例”。
3. **并存迁移 (Parallel & Migration)**：引入新方案时，旧的 File Queue 机制必须继续在线运行 30 天，提供 100% 的无缝回滚能力。
4. **高摩擦合流**：合流修改不变量文件必须伴随对 `EVIDENCE_MATRIX.md` 对应回归测试的重新覆写，使每一次例外都付出极高的“工程摩擦力”，防范由于顺手图省事而导致原则滑坡。

## 3. 失败博物馆 (The Ecosystem Museum)

* **设计定位**：文明的繁荣不仅因为记住了正确，更因为深刻记住了错误。
* **博物馆归档规范**：
  * 我们设立专用的 `/docs/museum/` 失败博物馆目录。
  * 任何被否决的提案、踏空的架构选型（如在论证中放弃的 Redis 方案、放弃的 SQLite 嵌入式存储设计、以及由于高耦合而被迫拆除的智能自愈代码），都必须以“负向案例”被完整保留，并挂载至 `MUSEUM.md` 索引中。
  * 当未来的继承者再次试图提出类似“为什么我们不搞一个内存缓存”的建议时，必须首先被强制导流阅读博物馆对应的失败报告，避免重复踩坑和无效的二次试错。

## 4. 探索者研究沙箱 (The Explorer Protocol)

为了在保持保守演化的同时不陷入停滞，启元生态建立“研究沙箱隔离制”：

* **探索者角色 (Explorer)**：探索者不具备系统的核心控制权，无权修改 main 分支代码，也无权向生产执行器发送指令。
* **沙箱隔离 (Sandbox-Research)**：探索者只能在隔离的 `research/` 实验沙箱内尝试最新的、高度不确定的模型、自愈算法或复杂的第三方网络中间件。
* **晋升通路**：任何探索性代码，若想合入系统主干，必须通过“三位一体证据验证”：不仅要补齐 Level 3 的 Architecture Budget，更要提供 Green 评级的回归测试（Evidence），证明其对核心不变量没有破坏性，才可由 Curator 核准晋升。

## 5. 监督者轮值与角色退役 (Role Rotation & Retirement)

* **谁来监督监督者 (Watcher Rotation)**：
  * 启元生态中没有任何角色（包括 Curator 策展者与 Historian 史官）拥有永久且绝对的解释权。
  * 实行**常驻守护进程（Daemons）的自校验轮值**：
    * `Curator` 的整理和审查规则，每天接受 `Evidence Daemon` 的自动化 Lint 脚本静态扫描，防止 Curator 自己成为过度设计者。
    * 每隔一个大版本发布，Curator 必须重新审视并回答：*“本角色在过去三个月内是否发起了至少一次减法（删除）动作？如果答案为否，本角色自身的权限应被强制收缩 20%。”*
* **角色的生命周期与退休 (Role Retirement)**：
  * 所有的治理角色和 Daemon 都不应该永久常驻。
  * 当系统进入高度收敛状态、连续 12 个月零安全漏洞且零契约冲突时，`Curator Daemon` 应该转入“休眠/半退役”状态，降低轮询频率，甚至在触发特定规则后彻底 Archived 归档，防止治理机制本身退化为冗余的行政官僚系统。

## 6. 终局遗产协议 (The Legacy Protocol)

* ** Legacy 宣告条件**：
  * 当启元生态完成其历史使命，或由于底层模型架构彻底转向、不再需要物理微执行器时，QY-EXEC 将启动《终局遗产冻结协议》。
* **终局冷冻程序 (Ecosystem Freezing)**：
  * 停止一切代码演变（Lock API），归档所有开发分支，解除所有常驻守护 Daemons，仅保留最简且绿色的 `/dist/` 物理运行态。
* **留给后世的唯二真理**：
  * 如果有一天 QY-EXEC 的代码全部失效，它希望未来的软件继承者依然能够从这个仓库里带走唯二的两条工程传统：
    1. **工程克制**：不要因为有能力增加功能就去无序堆积，真正的优雅是当功能过时时，可以将其无痛且干净地剥离（Deletability by Design）。
    2. **物理强制**：不要相信人的道德规范与自觉承诺，去构建自动化测试和 API 物理边界。**文档表达共识，测试验证共识，代码强制共识。**
