# 启元体系演化宪法 (The Evolution Constitution & Conservative Growth)

> **宪法前言 (Constitutional Intent)**
> 任何生物和文明的延续，本质上面临的是同一个终极命题：**如何在多变的环境中改变自己，同时又不会在剧变中迷失自己。**
> 备份是完全复制，而遗传是选择性保留。
> QY-EXEC 与整个启元多活生态并非为了满足短期功能堆叠而生，而是为了在 5 至 10 年的时间尺度上，实现跨越模型换代、跨越开发语言、跨越人类看护人员轮换的 **“连续文明（Continuous Civilization）”**。
> 本宪法确立了启元生态不可妥协的演化法则，保障体系在追求“保守演化（Conservative Evolution）”的同时，维护系统基因的一致性与生存自护能力。

## 1. 启元生态三维组织模型 (Organization & Responsibility Model)

启元不应被视为单一的大型 AI，而是由拥有严格职责边界与权限解耦的家族群体构成的**生态组织**。

```
                    人类 CEO / 外部世界
                           │
                    外部桥 (External Bridge)
                           │
      ┌────────────────────▼────────────────────┐
      │             治理层 (Governance)          │
      │             - 制定原则、核准演化         │
      └────────────────────┬────────────────────┘
                           │
         ┌─────────────────┴─────────────────┐
         │                                   │
   战略规划层 (Planner)                知识与记忆层 (Knowledge)
   - 调度资源、分拆任务、流程组合       - 归纳历史、长期记忆、RAG 注入
   - 成员: Planner Family              - 成员: Memory Family
         │                                   │
         └─────────────────┬─────────────────┘
                           │
                           ▼
                    控制与执行层 (Execution)
                    - 指令翻译、物理沙箱、状态保活
                    - 成员: Executor Family (QY-EXEC)
                           │
                           ▼
                    底层物理工具链 (Tool)
                    - 单一职责、原子功能 (read/write/patch)
                    - 成员: Tool Family
```

### 1.1 组织成员职责与权限约束
1. **治理层 (Governance)**：
   * *职责*：定义生态底线、裁决设计分歧、裁定并核准重大变更协议。
   * *权限*：拥有修改不变量（Invariants）与宪法文本的最高特权，但其本身不包含任何底层物理执行代码。
2. **战略规划层 (Planner)**：
   * *职责*：对用户或治理层发布的宏大意图进行多级分拆与动态规划。
   * *权限*：可对各执行器下发指令，但**绝对禁止**直接操作系统物理硬件、读写敏感系统盘、或修改系统底层不变量。
3. **控制与执行层 (Execution - QY-EXEC)**：
   * *职责*：将战略规划层的任务意图原子化翻译，交由物理工具链执行，并回传强结构化、错误分层的执行报告。
   * *权限*：具备在物理沙箱内调用工具和执行系统命令行能力，但**绝对禁止**擅自更改任务的宏观策略目标、不承担长期记忆存储、无权越权干涉决策流。
4. **底层物理工具链 (Tool)**：
   * *职责*：单职责完成物理原子动作（例如覆盖一个文件）。
   * *权限*：仅在沙箱路径内具备极高操作特权，但其本身禁止产生任何规划分支逻辑、**绝对禁止**横向相互嵌套与双向调用。

## 2. 演化 7 大核心宪法不变量 (The Seven Laws of Conservative Evolution)

### Law 1: 基因隔离原则 (Gene Autonomy)
* 任何底层的执行实体（Executor）、规划实体（Planner）或存储实体，**绝对无权**在执行或自我重构过程中擅自修改系统的底层基因（`INVARIANTS.md` 与本宪法）。一切基因变动必须外置于治理层核准。

### Law 2: 演化必须完全可逆 (Guaranteed Reversibility)
* 所有的协议升级、API 变更和实体行为演进，必须原生支持零状态损失的**版本回滚（Rollback）**。如果一个演进方向在遭受突发物理异常时无法无损退回前一稳定版本，则该演进设计属于非法设计，不予部署。

### Law 3: 先并存、后替换 (Coexist Before Replace)
* 绝不允许破坏性的断层替代。任何核心组件（如 `patch` 工具算法、或新的任务 JSON 字段）在升级时，必须在系统中采用“新旧并存 -> 灰度验证 -> 降级兜底 -> 彻底淡出（Deprecated）”的平滑渐进演进路线。

### Law 4: 演化必须留下历史 (Immutable Evolutionary History)
* 系统所有的重大决策必须以追加形式记录于 **ADR.md**。严禁通过直接覆盖、抹除历史决策文件的形式来修改演化逻辑。每一次演化，必须在历史库中清晰、可被未来继承者理解地载明：*为什么这样做，以及为什么不那样做*。

### Law 5: 演化不依赖个人与临时状态 (No Human/Instance Dependence)
* 演化的成功标准是“新加入的看护人员或新的模型实例，仅通过阅读系统规范层文档（Level 1-3），即可 100% 毫无偏差地理解和维护当前状态”。任何带有“只有某人懂”或“必须依赖当前进程活体状态”的隐式功能，均视为系统级债务。

### Law 6: 组织结构对齐优于代码实现 (Organization Alignment First)
* 在为任何物理实体增加新工具（Tool）或增强 API 之前，必须在系统契约（Contract）中首先回答五个基本治理问题：*该能力属于哪一个层级？由谁负责、由谁维护、如何测试、如何监控、如何无损删除？*

### Law 7: 保守演化与设计克制 (Conservative Evolution Limit)
* **不要为了证明系统能够演化而演化。** 系统的演化频度与深度必须处于极度克制状态。每一次 Release 必须无情接受“可删除性核验（Gate 5）”：*如果未来要把这个新功能彻底从代码库里抽除，代价是否极低？* 任何为了追求“概念炫技”而导致核心重度膨胀的提案，直接予以驳回。

## 3. 演化风险分层机制 (Evolution Risk Categorization)

不同的技术与契约变更对系统的生存与传承带来不同维度的挑战。我们实行四大风险分层申报与核验制度：

| 变更类别 (Evolution Type) | 影响范围 (Impact) | 变更评级 (Risk) | 决策核准与治理门槛 (Review Requirement) |
| :--- | :--- | :--- | :--- |
| **Type A: 纯物理实现演化** | 如 Python 升级、Node.js 库热更新、代码极简重构。 | **低 (Low)** | 仅需跑通 `npm run lint` 与 `compile_applet` 自动化测试，确保 Invariants 无损即可合流。 |
| **Type B: 能力与工具链演化** | 如增加全新的物理原子 Tool、或微调工具元数据 yaml。 | **中 (Medium)** | 必须在 `v1.1_strategic_evolution.md` 中补充声明，且必须包含配套的“自动化安全攻击回归测试用例”（如穿越防御测试）。 |
| **Type C: 职责与通信契约演化** | 如修改 `task_schema.json` 字段、或在家族成员间调整流程。 | **高 (High)** | 必须对新旧版本在多活沙箱内执行向下兼容灰度验证，并由团队联签，详细更新 `contracts.md` 与 `ADR.md`。 |
| **Type D: 基因与设计哲学演化** | 如修改 `INVARIANTS.md` 圣神底线、或变更本宪法。 | **最高 (Critical)** | **终极战略审查。** 必须召开全生态委员会听证，书写专门的 ADR 深度论证为何不变量在此刻不再成立，核准后方可更改。 |
