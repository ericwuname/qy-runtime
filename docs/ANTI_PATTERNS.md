# 启元工程反面模式防线 (QY-EXEC Anti-Patterns Guide)

> **前言 (The Preventive Intent)**
> 历史的长河中，同样的工程悲剧会被不同的团队重复演练。
> 博物馆（MUSEUM.md）记录了具体的技术选型妥协，而本文件则是抽象提炼后的**“工程疫苗”**。
> 它规定了在启元传统中，**“绝对不要这样干”**的红线行径，以及违反后的物理特征和检测方法。任何新人在合并代码（PR）前，必须对照本清单自检。

---

## 1. 模式一：试图将执行器（Executor）“规划化”与“智能化”
* **反面行径 (The Anti-Pattern)**：在 Executor 底层调用轻量级 LLM，或者在执行逻辑中引入条件多路分支试探（如“补丁失败后，自动重写 patch 代码重新尝试”）。
* **产生的毒害 (The Harm)**：
  1. 破坏了“不变量 1”（Executor 永不参与规划），模糊了 Planner 与 Executor 的职责边界。
  2. 引入了推理的随机波动。同样的错误输入，多次运行产生不同的自愈路径，导致系统的“确定性重放能力”丧失。
* **物理检测方法 (The Detection)**：
  * **AST 级审查**：CI 静态分析器检测 Executor 目录中是否导入了任何 AI/LLM 相关的 SDK 库或 API 客户端。
  * **网络边界拦截**：在沙箱运行期间，监控出站网络流量。如果 Executor 发生任何向外部 LLM 域名（如 `generativelanguage.googleapis.com`）的 API 连接，立刻物理切断。

---

## 2. 模式二：将临时状态“常驻化”与“隐秘化”
* **反面行径 (The Anti-Pattern)**：将任务状态、执行过程中的变量保存在本地内存中的单例（Singleton）或常驻全局字典里，供后续任务隐式读取。
* **产生的毒害 (The Harm)**：
  1. 破坏了“不变量 3”（执行端彻底无状态化），在物理机上产生了无法通过重放任务重现的“隐藏状态（Hidden State）”。
  2. 产生状态悬挂。一旦物理容器意外重启，保存在内存中的变量全部丢失，导致任务逻辑彻底卡死在中间态。
* **物理检测方法 (The Detection)**：
  * **沙箱热启动对比测试**：运行回归测试 `test_cold_start_reproducibility`。先运行任务 A 改变其本地内存/临时空间，再运行完全相同的任务 A，断言两次执行产生的物理输出字节流哈希 100% 一致。

---

## 3. 模式三：安全拦截逻辑的“业务级内聚”
* **反面行径 (The Anti-Pattern)**：将安全边界检查（如拦截 `rm -rf` 或是路径逃逸 `..`）直接写在具体工具（如 `bash_tool`、`file_reader`）的业务逻辑代码深处。
* **产生的毒害 (The Harm)**：
  1. 产生安全死角。每当新增、修改或者用另一种语言重写工具时，都需要人工复制安全校验代码，只要有一处遗漏，物理系统立刻洞开。
  2. 无法热升级。安全规则的改动要求重新编译并发布具体的业务工具，增加了系统的脆弱性。
* **物理检测方法 (The Detection)**：
  * **混沌穿越测试**：在回归测试中通过统一的中间件入口传入带有越权特征的输入（如 `validate_safe_path("../etc")`），断言其 100% 在“前置过滤器”就被抛出异常，而不需要触及任何具体工具执行逻辑。

---

## 4. 模式四：描述性报告的“规范制度化”（描述与约束混淆）
* **反面行径 (The Anti-Pattern)**：在常驻守护进程（Daemons）产生的运行报告（Health Report）或日志中，定义新的约束和不变量规则；或者在不变量（Invariants）中写满“今日 CPU 占用率、昨晚网络延迟”等描述性监控数据。
* **产生的毒害 (The Harm)**：
  1. 角色职责混乱。导致新人无法区分“什么是客观的物理事实”（Descriptive）与“什么是主观的神圣底线约束”（Normative）。
  2. 增加不必要的认知负载，使规范文档膨胀为垃圾监控日志。
* **物理检测方法 (The Detection)**：
  * **审查分流**：任何常驻守护进程生成的自校验输出，只允许写入 `descriptive/` 目录或发送给 Prometheus 监控端，绝对禁止将其作为 Level 1 宪章进行合流。
